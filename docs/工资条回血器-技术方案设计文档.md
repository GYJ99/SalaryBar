## 3. 技术方案结论

首版建议采用：

- 开发语言：`Swift`
- UI 框架：`SwiftUI`
- macOS 顶部栏形态：`MenuBarExtra`
- 辅助能力：必要时混合 `AppKit`
- 本地存储：`UserDefaults` + `Codable`
- 时间与日历：`Foundation / Calendar / DateComponents`
- 系统通知：`UserNotifications`
- 开机启动：`ServiceManagement`
- 构建工具：`Xcode`
- 安装包产物：`.app` + `.dmg`

## 4. 为什么选择原生方案

### 4.1 不建议首版使用网页容器方案

不建议首版使用 `Electron`、`Tauri` 或 `WebView` 方案，原因如下：

- 你的目标不是 Web 页面，而是顶部栏常驻应用
- 顶部栏状态刷新需要轻量和稳定
- 原生方案更适合菜单栏交互、系统通知和开机启动
- 原生方案在 CPU、内存、系统观感上更符合“小工具”定位

### 4.2 SwiftUI + AppKit 混合的价值

纯 `SwiftUI` 可以完成大部分界面开发，但部分系统级能力仍建议保留 `AppKit` 接入点。

推荐做法：

- UI 层以 `SwiftUI` 为主
- 顶部栏宿主、窗口控制、某些兼容场景使用 `AppKit` 封装

### 6.1 架构原则

首版采用轻量但可扩展的分层架构：

- 表现层与计算层分离

## 10. 核心状态设计

### 10.1 顶部栏状态枚举

```swift
enum WorkState {
    case unconfigured
    case active
    case paused
    case offHours
}
```

### 10.2 页面展示态

不同状态映射到 UI：

- `unconfigured`：显示去设置入口
- `active`：显示实时增长金额

- `offHours`：显示未开工或今日总额

## 11. 收入计算方案

### 11.1 核心原则

所有收入展示统一基于“每秒收入”推导。

```text
每秒收入 = 计薪收入 / 可计薪总秒数
累计收入 = 每秒收入 × 已计薪秒数
```

### 11.2 月薪模式换算

首版建议支持 3 种规则：

1. 按当月工作日换算
2. 按固定月工时换算
3. 按 21.75 天规则换算

推荐定义：

```swift
enum SalaryConversionRule: Codable {
    case byWorkdaysPerMonth(workdays: Decimal, dailyHours: Decimal)
    case byFixedMonthlyHours(hours: Decimal)
    case byStandardMonthlyDays(dailyHours: Decimal)
}
```

### 11.3 时薪模式换算

```text
每秒收入 = 时薪 / 3600
每分钟收入 = 时薪 / 60
每小时收入 = 时薪
```

### 11.4 精度方案

首版建议：

- 业务计算使用 `Decimal`
- 中间态避免转成 `Double`
- 展示时统一通过 `NumberFormatter` 格式化

原因：

- `Decimal` 更适合货币计算
- 长时间按秒累计时误差更可控

## 12. 计时引擎设计

### 12.1 为什么不能只靠“每秒 +1”

如果单纯依赖一个固定 `Timer` 每秒累加，会出现以下问题：

- 应用休眠后计时不准
- 系统卡顿时累计偏差
- 后台调度延迟导致金额不准

因此应采用“基于真实时间差重新结算”的设计。

### 12.2 推荐算法

每次 tick 执行：

1. 读取当前时间 `now`
2. 获取上次结算时间 `lastTickAt`
3. 计算区间 `[lastTickAt, now]`
4. 判断这段时间内哪些秒属于可计薪时段
5. 累加有效秒数对应的收入
6. 更新 UI 和本地快照

### 12.3 可计薪判断条件

某个时间点被认为可计薪，需同时满足：

- 已完成基础配置
- 当前日期为工作日
- 当前时间在工作时间段内
- 当前时间不在休息时段内
- 当前状态不是手动暂停

### 12.4 跨天处理

当 `now` 与 `currentDay` 不在同一天时：

- 将“今日已赚”切换到新的一天
- 清空今日累计
- 保留配置不变
- 若新的一天当前处于工作时间，可自动继续累计

### 12.5 休眠与唤醒

唤醒后不做简单“补 N 秒”，而是：

- 读取休眠前最后时间点
- 读取当前时间
- 按真实时间段重算可计薪秒数

这样可以自动跨越休眠、锁屏、切桌面等场景。

## 13. Timer 与刷新策略

### 13.1 UI 刷新目标

用户感知上是“每秒跳动一次”，但内部可以做更稳妥的时间驱动。

### 13.2 推荐实现

- 使用 `DispatchSourceTimer` 或 `Timer.publish`
- 节拍建议 1 秒
- 真正累计逻辑基于 `Date` 差值，不依赖 tick 次数

### 13.3 优化点

- 顶部栏标题仅在展示值变化时更新
- 面板关闭时仍需维持核心累计，但可减少非必要重绘
- 解锁通知只在状态变化瞬间触发一次

## 14. 目标进度计算设计

### 14.1 排序规则

- 目标按金额升序排序
- 如用户手动排序，则按显示顺序渲染，但解锁判断仍基于金额

### 14.2 当前目标判定

当前目标 = 金额大于当前累计值的最小目标项。

### 14.3 进度条算法

若当前目标存在：

```text
progress = 当前累计金额 / 当前目标金额
```

若当前目标不存在：

- 表示全部目标已完成
- 进度条显示 100%

### 14.4 解锁事件

当某目标从“未解锁”变为“已解锁”时：

- 更新详情列表状态
- 触发一次本地通知
- 记录已通知标记，避免重复提醒

## 15. 本地存储设计

### 15.1 首版建议

首版不需要引入数据库，使用：

- `UserDefaults` 存储轻量配置
- `Codable` 编解码结构体

### 15.2 存储项建议

- `salary_config`
- `work_schedule_config`
- `goal_items`
- `display_preferences`
- `notification_preferences`
- `runtime_snapshot`

### 15.3 Repository 封装

不要在 View 或 ViewModel 里直接操作 `UserDefaults`。

推荐封装：

- `SettingsRepository`
- `RuntimeSnapshotRepository`

优势：

- 更易测试
- 后续切换到 `SwiftData` 或文件存储时改动更小

## 16. 通知方案

### 16.1 通知类型

首版支持：

- 新目标解锁通知
- 开工提醒
- 下班提醒

### 16.2 技术实现

- 使用 `UNUserNotificationCenter`
- 首次相关操作时申请权限
- 通知内容由业务层产出，平台层负责分发

### 16.3 去重策略

- 同一目标同一天不重复通知
- 状态未变化不重复发

## 17. 开机启动方案

### 17.1 技术实现

使用 `ServiceManagement` 管理开机启动。

### 17.2 行为要求

- 设置页可开启/关闭
- 开机启动后直接进入顶部栏常驻模式
- 不强制弹出主窗口

## 18. 窗口与生命周期设计

### 18.1 窗口类型

- 顶部栏弹出面板
- 设置窗口
- 首次引导窗口

### 18.2 生命周期要求

- 关闭设置窗口时应用继续运行
- 用户点击“退出”才真正结束进程
- 再次点击顶部栏时应复用现有面板状态

## 19. 异常与边界场景处理

### 19.1 未配置

- 顶部栏显示未配置提示
- 弹出面板展示引导态
- 禁止进入正常计时

### 19.2 非法时间配置

例如：

- 上班时间晚于下班时间
- 午休区间越界
- 工作时长为 0

处理方式：

- 设置页直接拦截
- 阻止保存
- 清晰提示错误原因

### 19.3 系统时间变化

若系统时间被用户手动修改：

- 重新校正 `lastTickAt`
- 必要时重新结算当日累计
- 避免出现负时长累计

### 19.4 时区变化

首版按本机当前时区运行。

若时区变化：

- 以新时区重新判断工作日和工作时段
- 同一天累计是否重算，可先采用“保留当前已累计值”的保守策略

### 19.5 休眠唤醒

- 监听系统唤醒事件
- 触发一次立即结算
- 强制刷新顶部栏显示
